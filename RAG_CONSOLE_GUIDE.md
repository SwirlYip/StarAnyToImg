# RAG 控制台日志使用指南

## 🎯 功能说明

星澜智能绘图已在前后端集成了完整的 RAG（知识库检索）日志系统。你可以在浏览器的 F12 开发者工具控制台中查看详细的知识库匹配信息。

---

## 🚀 如何查看日志

### 步骤 1：打开 F12 开发者工具

- **Windows/Linux**: 按 `F12` 或 `Ctrl + Shift + I`
- **Mac**: 按 `Cmd + Option + I`

### 步骤 2：切换到 Console（控制台）标签

点击顶部的 "Console" 标签，进入控制台。

### 步骤 3：发起生成请求

在前端页面输入描述（如"江南水乡风景"），点击"✨ 开始生成"按钮。

### 步骤 4：查看 RAG 日志

控制台中会自动展开分组日志，显示以下信息：

```
▼ 🧠 LLM 增强结果
  📝 用户描述: "江南水乡风景"
  ✅ 正向提示词: "masterpiece, best quality, 8k, high resolution, ..."
  🚫 反向提示词: "blurry, low quality, distorted, ..."
  ⚙️  生成参数: {步数: 30, CFG: 6, 种子: 196362294227}
  ▼ 🎯 RAG 知识库检索
    ✅ 匹配到 5 个知识库类别:
       1. 风景·江南水乡 (分数: 8)
       2. 风景·森林秘境 (分数: 4)
       3. 风景·海边日出 (分数: 4)
       4. 风景·雪山冰川 (分数: 4)
       5. 风景·沙漠戈壁 (分数: 4)
    ▼ 📊 Token 优化
       原方案: 32066 tokens
       检索后: 2513 tokens
       节省: 29553 tokens
       节省率: 92.2%
```

---

## 📊 日志字段说明

### 🎯 RAG 知识库检索

| 字段 | 说明 |
|------|------|
| **matchedCount** | 匹配到的知识库类别数量 |
| **categories** | 匹配的类别列表，包含类别名和匹配分数 |
| **score** | 匹配分数，数值越高表示越相关 |
| **tokenOptimization** | Token 优化统计信息 |

### 📊 Token 优化

| 字段 | 说明 |
|------|------|
| **original** | 原全量注入方案使用的 tokens（32066）|
| **reduced** | 智能检索后实际使用的 tokens |
| **saved** | 节省的 tokens 数量 |
| **savedPercent** | 节省百分比 |

## 🔍 如何判断 RAG 是否生效

### ✅ 生效的标志

1. **匹配分数合理**：
   - 精准匹配（如"江南水乡"）→ 第 1 名分数 ≥ 8
   - 相关匹配 → 分数 2-6
   - 无关匹配 → 不会出现

2. **Token 优化显著**：
   - 节省率 ≥ 90% 说明检索器正常工作
   - 检索后 tokens 通常在 500-2500 之间

3. **提示词包含知识库特征**：
   - 江南水乡 → `misty rainy day, aged white walls, moss`
   - 古风汉服 → `Hanfu, traditional Chinese clothing, embroidery`
   - 日系治愈 → `Japanese anime style, healing vibe`

### ❌ 未生效的标志

1. **日志显示没有匹配**：
   ```
   ⚠️ 没有匹配的知识库类别，使用通用规则
   ```

2. **提示词泛化且通用**：
   - 只有 `masterpiece, best quality, 8k`
   - 没有类别专属关键词

3. **Token 优化不正常**：
   - 检索后 tokens 仍然很高（>10000）
   - 或者节省率 < 80%

## 🛠️ 故障排查

### 问题：看不到 RAG 日志

**原因 1：前端缓存**
```bash
# 解决：硬刷新页面（Ctrl + Shift + R 或 Cmd + Shift + R）
```

**原因 2：控制台日志未展开**
```bash
# 解决：点击 ▼ 箭头展开分组日志
```

**原因 3：网络错误**
```bash
# 检查：Network 标签中 API 请求是否成功
# 查看：Response 中是否包含 enhanced.rag 字段
```

### 问题：匹配到错误的类别

**排查步骤：**
1. 检查用户描述是否准确（如"江南水乡"不要写成"江南风光"）
2. 知识库中是否有该类别（查看 `knowledge.json`）
3. 匹配分数是否合理（最高应该是最相关的）

### 问题：Token 节省率过低

**正常范围：**
- 精准匹配 → 98% 节省（580 tokens）
- 通用匹配 → 92-95% 节省（1500-2500 tokens）

**异常情况：**
- 节省率 < 80% → 检索器可能有问题，联系开发者

## 📈 监控建议

### 日常关注

每次生成时，快速检查：

1. **匹配类别数量**：1-5 个为正常
2. **最高分分数**：≥ 8 为精准匹配
3. **Token 节省率**：≥ 90% 为正常
4. **正向提示词**：是否包含知识库关键词

### 测试验证

定期测试以下场景：

1. **精准匹配**：`江南水乡`, `古风汉服`, `赛博朋克`
2. **泛匹配**：`风景`, `人像`, `动漫`
3. **无匹配**：`完全无相关的描述`

## 💡 进阶技巧

### 1. 过滤特定日志

如果你只想看 RAG 日志，可以在控制台输入：

```javascript
// 清空控制台
console.clear()

// 发起请求后，只展开 RAG 日志
```

### 2. 查看历史日志

在页面多生成几次后，向上滚动查看历史上的 RAG 匹配记录。

### 3. 对比不同描述

用不同的描述生成，对比：
- 匹配的类别是否不同
- Token 节省率是否有差异
- 提示词是否体现出类别差异
